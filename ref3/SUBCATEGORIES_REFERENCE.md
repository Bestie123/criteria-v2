### architecture, ai-criteria, tools

**Подкатегории обычно не используются** - принципы и инструменты достаточно конкретны и не требуют дополнительной классификации.

## Когда создавать новую подкатегорию?

✅ **Создавай новую подкатегорию когда:**
- Критерий применяется по-разному в разных контекстах одной категории
- Нужно уточнить специфику применения
- Критерий относится к устоявшейся подобласти
- Подкатегория будет использована в нескольких критериях

❌ **НЕ создавай новую подкатегорию когда:**
- Критерий универсален для всей категории
- Контекст очевиден из текста критерия
- Подкатегория будет использована только один раз
- Можно использовать существующую подкатегорию

## Примеры использования

### Пример 1: Диаграмма зависимостей
```json
{
  "categories": ["quality"],
  "contexts": {
    "quality": {
      "subcategories": ["Архитектурная целостность"],
      "descriptions": {
        "Архитектурная целостность": "Использовать диаграммы для визуализации архитектуры"
      }
    }
  }
}
```

### Пример 2: Множественные варианты решений
```json
{
  "categories": ["quality", "prompts"],
  "contexts": {
    "quality": {
      "subcategories": ["Best Practices & Code Review"],
      "descriptions": {
        "Best Practices & Code Review": "Рассматривать несколько вариантов решений"
      }
    },
    "prompts": {
      "subcategories": ["Конструктивная критика (шаблоны)"],
      "descriptions": {
        "Конструктивная критика (шаблоны)": "Сравнивать несколько подходов"
      }
    }
  }
}
```

### Пример 3: Без подкатегорий
```json
{
  "categories": ["architecture"],
  "contexts": {
    "architecture": {
      "subcategories": [],
      "descriptions": {
        "null": "Модульность архитектуры - универсальный принцип"
      }
    }
  }
}
```
